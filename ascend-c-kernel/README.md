# Ascend C TQ3 Dequant Kernel (In Progress)

## Status

The custom Ascend C kernel for TQ3 dequantization is **work in progress**. We have:

- ✅ Generated the msopgen project skeleton
- ✅ Written the operator descriptor (`op_desc.json`)
- ✅ Filled in the host-side operator code (`tq3_dequant_host.cpp`)
- ⏳ The device kernel body (`tq3_dequant_kernel.cpp`) is a **template with an empty body**
- ❌ `ccec` compilation of `kernel_operator.h` fails — the `__aicore__` attribute macro expands to `[aicore]` which the compiler doesn't recognize as a valid attribute

## Problem

The Ascend C kernel compilation currently fails at:

```bash
ccec --c++17 -std=c++17 -target=ascend910 -mcpu=tsdaa \
     -I ${ASCEND_TOOLKIT_HOME}/aarch64-linux/tikcpp/tikcfw/ \
     -I ${ASCEND_TOOLKIT_HOME}/aarch64-linux/ascendc/include/ \
     kernel_operator.h
```

Error:
```
warning: 'aicore' attribute directive ignored [-Wattributes]
int32_t main() __attribute__((aicore));
```

The `__aicore__` macro expands to `__attribute__((aicore))` but the compiler does not recognize the `aicore` attribute at this stage. The proper path uses the full msopgen → cmake → opbuild flow.

## What We Learned

1. **msopgen** can generate a complete custom operator project skeleton
2. The **host-side** code (operator registration, ACLNN API) compiles and links successfully via cmake
3. The **device kernel** requires the full Ascend C toolchain (`opbuild`) — directly invoking `ccec` on `kernel_operator.h` is not the correct approach
4. Device kernel `.o` files are generated by `opbuild` as part of the cmake build in the msopgen project
